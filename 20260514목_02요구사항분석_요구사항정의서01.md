## 1. 문서 목적

본 문서는 NeighborTrade 서비스의 주요 기능 요구사항을 정의한다. 시스템 구현, 화면 설계, 테스트 케이스 작성 및 운영 기준 수립을 위한 공통 기준으로 활용한다.

---

## 2. 범위

본 요구사항 정의서는 NeighborTrade의 초기 MVP 범위에 속하는 다음 기능을 포함한다.

- 회원가입·로그인·로그아웃
- 동네 선택 및 GPS 기반 동네 인증
- 열람 동(지역) 설정 및 변경
- 판매글 목록 조회·검색·상세 조회
- 판매글 등록·수정·삭제
- 거래 요청·진행·완료
- 거래 후기 작성·조회
- 마이페이지 조회
- 관리자 운영 기능

2차 MVP 또는 이후 단계로 분류되는 기능은 다음과 같다.

- 동네생활 게시판
- 채팅
- 신고 접수

## 4. 용어 정의

- 열람 동: 사용자가 판매글 목록을 조회할 때 기준이 되는 동네. 로그인 유무와 관계없이 설정 가능.
- 인증 동네: 사용자가 GPS 인증을 통해 본인 거주지를 확인한 행정동. 인증 회원 권한 부여에 사용.
- 판매글: 중고거래 게시물.
- 거래 요청: 구매자가 판매자에게 거래 의사를 전달하는 단계.
- 거래 후기: 거래 완료 후 거래 참여자가 남기는 평가/댓글.

## 5. 기능 요구사항

### 5.1 회원 관리

#### 5.1.1 회원가입 (UC-01)

- 비회원은 로컬 계정으로 회원가입을 할 수 있어야 한다.
- 회원가입 필수 정보: 아이디(username), 비밀번호(password), 이메일(email), 닉네임(nickname).
- 입력값에 대해 유효성 검사와 중복 검사를 수행해야 한다.
- 중복 아이디 또는 중복 이메일이 입력된 경우 오류 메시지를 출력해야 한다.
- 가입 완료 시 `member` 레코드가 생성되고, 로그인 가능한 상태가 되어야 한다.

#### 5.1.2 로그인·로그아웃 (UC-02)

- 회원은 아이디/비밀번호 또는 외부 인증(Kakao/Naver)으로 로그인할 수 있어야 한다.
- 로그인 성공 시 인증 세션을 생성한다.
- 로그아웃 요청 시 세션을 무효화하고 공개 화면으로 이동시켜야 한다.
- 로그인 실패 시 비밀번호 불일치, 계정 미존재, 소셜 로그인 오류 등 적절한 오류 메시지를 보여야 한다.

### 5.2 동네 인증 및 지역 관리

#### 5.2.1 동네 선택 후 GPS 인증 (UC-03)

- 로그인한 회원은 동네 목록에서 인증하려는 행정동을 선택할 수 있어야 한다.
- 인증 과정에서 브라우저 위치 정보 허용을 요청한다.
- 서버는 제출된 GPS 좌표를 기준으로 행정동 경계 또는 반경 검증을 수행해야 한다.
- 인증에 성공하면 회원 정보에 `verified_neighborhood_id`, `local_verified`, `role` 값이 갱신되어야 한다.
- 인증 이력은 저장되어야 하며, 실패 시 사유 안내 후 재시도할 수 있어야 한다.
- 위치 권한 거부 또는 좌표 수신 실패 시 오류 처리와 안내가 필요하다.

#### 5.2.2 인증 동네 확인 (UC-04)

- 로그인한 회원은 자신의 인증 동네와 인증 상태를 확인할 수 있어야 한다.
- 마이페이지 또는 동네 인증 확인 화면에서 인증 여부와 동 이름을 표시해야 한다.

#### 5.2.3 열람 동 설정·변경 (UC-05)

- 모든 사용자(비회원 포함)는 열람 동을 설정하거나 변경할 수 있어야 한다.
- 열람 동 정보는 쿠키 또는 URL 파라미터(`nh`)에 저장될 수 있어야 한다.
- 선택된 열람 동은 헤더의 지역 칩에 표시되고, 판매글 목록에서 강조되어야 한다.
- 전체 동네 보기 옵션을 통해 열람 동 선택을 해제할 수 있어야 한다.
- 동 목록 API 호출 실패 시 안내 메시지를 제공해야 한다.
- 로그인 여부와 관계없이 지역 변경 기능을 제공해야 한다.

### 5.3 게시물 검색·조회

#### 5.3.1 판매글 목록·검색 조회 (UC-06)

- 모든 사용자는 판매글 목록을 조회할 수 있어야 한다.
- 검색어, 거래 가능 필터 등 조건을 적용하여 목록을 필터링할 수 있어야 한다.
- 열람 동이 선택된 경우 해당 동에 속하는 판매글 또는 관련 글을 강조해서 보여야 한다.
- 열람 동이 해제된 경우 전체 동네 목록으로 보여야 한다.

#### 5.3.2 판매글 상세 조회 (UC-07)

- 모든 사용자는 판매글 상세 페이지를 열 수 있어야 한다.
- 상세 페이지에는 제목, 내용, 이미지, 가격, 판매자 정보, 동네 정보 등이 표시되어야 한다.
- 조회수 증가, 문의 버튼 등 부가 기능은 구현 정책에 따라 추가할 수 있다.

### 5.4 판매글 작성 및 수정

#### 5.4.1 판매글 등록 (UC-08)

- 동네 인증이 완료된 회원(A3)만 판매글을 작성할 수 있어야 한다.
- 판매글 작성 시 제목, 카테고리, 가격, 상태, 상세 설명, 이미지 등 입력 항목을 제공해야 한다.
- 입력값 검증 후 `product_post`가 생성되어야 한다.
- 미인증 회원이 접근 시 접근 거부 또는 인증 안내 메시지를 표시해야 한다.

#### 5.4.2 판매글 수정·삭제 (UC-09)

- 판매글 작성자는 자신의 글만 수정하거나 삭제할 수 있어야 한다.
- 수정 폼은 기존 내용을 불러와서 변경 후 저장할 수 있어야 한다.
- 삭제 시 확인 절차를 거쳐야 한다.
- 작성자 본인이 아니면 수정 및 삭제 기능을 차단해야 한다.

### 5.5 거래 관리

#### 5.5.1 거래 요청·진행·완료 (UC-10)

- 거래 참여자는 거래 요청을 주고받을 수 있어야 한다.
- 판매글 상태가 거래 가능이어야 거래 요청이 가능하다.
- 판매자는 요청을 수락하거나 거절할 수 있어야 한다.
- 거래 완료 또는 취소 절차를 제공해야 한다.
- 거래 상태 변화는 관련 화면에 반영되어야 한다.

#### 5.5.2 거래 후기 작성·조회 (UC-11)

- 거래가 완료된 후 참여자는 후기를 작성할 수 있어야 한다.
- 후기는 거래 이력에 연결되어 저장되어야 한다.
- 기타 사용자는 본인이 받은 후기를 조회할 수 있어야 한다.
- 후기 작성 조건(거래 완료 등)은 정책에 따라 검증되어야 한다.

### 5.6 개인 정보·마이페이지

#### 5.6.1 마이페이지 조회 (UC-12)

- 로그인한 회원은 마이페이지에서 계정 요약, 인증 상태, 매너 정보, 활동 링크를 확인할 수 있어야 한다.
- 마이페이지는 회원별 기본 정보와 현재 상태를 표시해야 한다.

### 5.7 운영 및 관리자 기능

#### 5.7.1 관리자 운영·대시보드 (UC-16)

- 관리자(A4)는 신고 처리, 회원 관리, 동네 관리, 게시물 관리를 수행할 수 있어야 한다.
- 관리자 전용 UI 또는 페이지가 제공되어야 한다.
- 관리자 접근은 관리자 권한이 있는 사용자로 제한해야 한다.

## 6. 비기능 요구사항

- 반응형 UI를 적용해 데스크톱과 모바일 환경에서 사용 가능해야 한다.
- 위치 정보 접근은 HTTPS 환경에서만 허용한다.
- 로그인 세션은 보안 세션 쿠키로 관리해야 한다.
- 회원 개인정보와 인증 동네 정보는 적절히 보호되어야 한다.
- 동네 인증 검증은 성능을 고려해 효율적으로 처리되어야 한다.
